## TOGAF: The INS Framework

### Enterprise Architecture — Right-Sized for INS

This engagement follows TOGAF (The Open Group Architecture Framework) — the world's most widely adopted EA standard — but fast-tracked and right-sized for a mid-market professional services firm. TOGAF's Architecture Development Method (ADM) provides the rigour; our approach removes the bureaucracy.

In a large enterprise, TOGAF typically means a dedicated architecture practice of 10-50+ architects, multi-month engagement cycles, formal review boards, and thousands of static artefacts in specialised tooling. That approach would be disproportionate for an organisation of INS's scale (~800 people, ~£100M turnover, primarily Microsoft stack). The framework would become the work.

### How We Apply TOGAF at INS

Every ADM phase is covered. Nothing is skipped. But the method is compressed, the governance is lean, and the tools are modern:

| TOGAF Phase | What It Covers | How We Deliver It at INS |
|-------------|---------------|--------------------------|
| **Preliminary** | Framework, principles, tools, governance | EA Framework Summary + ontology tooling + OAA v6 quality standard — already in place |
| **A: Architecture Vision** | Scope, stakeholders, high-level vision | 1-day facilitated workshop — decisions in a room in one day because the decision-makers are all present |
| **B: Business Architecture** | Business capabilities, processes, organisation | BSC strategy cascade (16 objectives, 5 pillars) + process catalogue (top 20) + stakeholder interviews |
| **C: Information Systems** | Data architecture, application architecture | Enterprise Graph specification + ontology framework (23 ontologies) + data source mapping |
| **D: Technology Architecture** | Infrastructure, platforms, security | 5 snapshot audits (ALZ, O365, PP, TP, RCSG — 33 artefacts) + 5 ADRs |
| **E: Opportunities & Solutions** | Options, work packages, build vs. buy | Three pillars framework + 14 PPM epics + dependency analysis |
| **F: Migration Planning** | Roadmap, sequencing, transition states | Capability roadmap (4/12/24 months) + Phase 2 Gantt + programme design |
| **G: Implementation Governance** | Conformance, oversight, change control | Steering committee (monthly) + PR-based artefact governance + OAA v6 quality gates |
| **H: Change Management** | Architecture evolution, versioning | Ontology versioning in GitHub + graph evolution + continuous audit monitoring |

### What Makes the INS Approach Different

**Ontology-driven, not document-driven.** The architecture exists as machine-readable models connected in a graph — not as static documents that become outdated within months. When one model changes, cross-references are immediately visible. Cross-domain traceability is a graph query, not a meeting.

**AI-augmented at every stage.** Architecture diagrams generated in minutes (Mermaid), ADRs drafted with AI-assisted options analysis, compliance cross-mappings automated through ontology relationships, BSC alignment validated automatically. Work that would take a team of analysts weeks is done in hours.

**Lean governance that fits the team.** A small steering committee, direct access to leadership, PR-based artefact review, and Architecture Decision Records — not formal review boards with multi-week approval cycles. Decisions can be made in a room in an afternoon.

**Platform-agnostic but Microsoft-aware.** The architecture leverages the deep integration benefits of the Microsoft stack (Azure, M365, Entra ID, Purview, Copilot, Power Platform) while keeping the Enterprise Graph on open standards (JSON-LD, schema.org) and selecting AI capabilities on merit, not vendor affinity.

### EA Maturity Trajectory

| Milestone | Maturity Level | What It Means |
|-----------|---------------|---------------|
| **Today** | Level 1 — Initial | No formal EA, ad hoc technology decisions, no unified data model, ~60% manual processes |
| **May 2026** (after this engagement + Phase 2) | Level 2 — Under Development | Architecture process defined, core artefacts produced, governance operational, key decisions documented |
| **Feb 2027** | Level 3 — Defined | Architecture practice consistent, Enterprise Graph populated, all audits at v2+, architecture informs all technology decisions |
| **Feb 2028** | Level 4 — Managed | Architecture value demonstrated through BSC metrics, continuous assurance operational, manual processes <=20% |

This trajectory is realistic and measurable. Moving from Level 1 to Level 2 in Phase 2 is achievable with a lean team and AI augmentation.

See **EA-TOGAF-Rationalisation-v1.md** for the full TOGAF definition, phase-by-phase coverage matrix, and detailed maturity assessment.

## 4. Approach

### Phase A: Pre-Workshop

- Review and synthesise all existing audit artefacts (33+ artefacts across ALZ, O365, PP, TP, RCSG)
- Review strategy documents: Context Summary v3.1, Wave 1 Priorities, Extended Portfolio, BSC framework
- Prepare workshop materials: visuals, demo environment, pre-reading pack

- Stage the ontology visualiser with unified registry for live demo

### Phase B: Workshop

- Facilitate the 1-day EA Vision & Roadmap Workshop (EA-PPM-WS-2026-001)
- Session 1: The EA Model (capability layers, Enterprise Graph, TOGAF, live demo)
- Session 2: Where We Are (audits mapped to EA model, manual process problem, gap summary)
- Session 3: The Vision (three pillars framework, capabilities per pillar)
- Session 4: The Roadmap (strategic themes, Phase 2 commitments, constraints)
- Session 5: Agreement (decisions, design assignment commission, actions)
- Real-time capture of decisions and actions in markdown

### Phase C: Design Assignment

- **Discover**: Deep-dive into current state, stakeholder interviews, process catalogue build, gap analysis synthesis
- **Design**: EA architecture across 4 layers, Enterprise Graph specification, AI service catalogue, technology ADRs, agentic layer architecture
- **Specify**: Programme structure, capability roadmap, BSC implementation map, process automation roadmap, final review and presentation
